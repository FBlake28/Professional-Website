This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## What this is

A single-page personal resume/portfolio website for Frederick Blake (`index.html`), styled with Tailwind CSS. There is no build tooling, package manager, bundler, linter, or test suite in this repo (no `package.json`) — the page is plain static HTML/CSS meant to be opened directly in a browser or served as-is.

The site is authored primarily through **Pinegrow**, a visual/no-code desktop web builder, not through a code-first workflow:

- `pinegrow.json` is the Pinegrow project file (registered frameworks/libraries: Tailwind, Flowbite, Pinegrow Interactions, SVG lib, CSS grid, image overlay, etc.). It tracks Pinegrow's own project state separately from the HTML.
- `tailwind_theme/tailwind.css` is a fully compiled Tailwind v3.4.3 output (all base/utility styles plus a Google Fonts `@import` for Ubuntu and IBM Plex Serif) generated by Pinegrow — it is not a source config file, and there is no `tailwind.config.js` or build step in this repo to regenerate it from source.
- The `<script>` block in `<head>` and the `data-pg-ia`, `data-pg-id`, `data-name` attributes throughout `index.html` are Pinegrow's auto-generated "Interactions" wiring (GSAP-based show/hide/animation logic tied to breakpoints). Treat these as generated output — hand-editing them is fragile and easy to break silently, since Pinegrow expects to own this markup.
- `index.html` references `pgia/lib/pgia.js`, but no `pgia/` directory exists in this repo. That script isn't checked in; it's expected to be supplied by Pinegrow on publish/export. Don't "fix" this by inventing a replacement file without asking — it may simply be how Pinegrow's publish step fills the gap.
- `_pgbackup/` (repo root) and `tailwind_theme/_pgbackup/` contain Pinegrow's automatic timestamped snapshots (`index_<timestamp>.html`, `pinegrow_<timestamp>.json`, `tailwind_<timestamp>.css`). These are backup history, not part of the live site — don't edit or treat them as source of truth.

Editing the raw Tailwind utility classes and text content in `index.html` directly is safe and is plain HTML — no build step is required to see changes, just reload the file in a browser. The risk is specifically around the Pinegrow-managed interaction attributes and the fact that reopening/resaving the file in the Pinegrow desktop app can regenerate or overwrite hand-made structural edits.

## Structure

`index.html` is a single page with in-page anchor navigation. Section IDs match the nav links in the header: `#Home`, `#Bio`, `#Exp` (Experience), `#Edu` (Education), `#Projects`, `#Skills`, and `#contact` (footer). All content — bio text, work experience, education, project list, skills/certs, contact links — lives in this one file; there are no templates, partials, or includes.

Image/PDF assets referenced by `index.html` (profile photos, project images, published articles) live at the repo root alongside the HTML and are linked by relative path.

## Working with this repo

- There's nothing to install, build, lint, or test. To preview changes, open `index.html` in a browser.
- Prefer editing `index.html` and `tailwind_theme/tailwind.css` directly for content/style changes; avoid modifying `pinegrow.json` unless the change is specifically about Pinegrow's registered frameworks/libraries.
- Don't clean up or "fix" the `_pgbackup` directories — they're Pinegrow-managed history, not build artifacts.
